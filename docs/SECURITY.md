# Security notes & known limitations

## Magic-link token appears in HTTP access logs (known limitation, not fixed)

The client magic-link is `GET /r/<token>`. The HMAC token is a **URL path
component**, so it is recorded by anything that logs request lines:

- the uvicorn access log (`INFO: ... "GET /r/<token> HTTP/1.1"`)
- any reverse proxy / load balancer in front (nginx, etc.)
- the client's browser history and the `Referer` header on outbound
  requests from the page

This is an **inherent property of token-in-URL magic links**, not a code
defect. The application's own structured logger is clean — verified by
`tests/web/test_app_route.py::test_RT2_*` and by the Slice-3 smoke
(`magic_link_rejected reason=… context=…`, token never present). Only the
transport/access layer captures the token.

### Why it is accepted (for now)

- Tokens are short-lived: TTL is hard-capped at **7 days** (and the SB
  typically issues 1–2 day links). Exposure window is bounded.
- An expired link is cheaply regenerated by the SB; no long-lived
  credential is at stake.
- Moving the token off the URL path (POST body / header) would break the
  core UX — "click a link in an email". That tradeoff was made
  deliberately at the Slice-3 seam review.

### Recommended production mitigations (deployment-time, not code)

- Disable or scrub the access log for the `/r/*` route at the uvicorn /
  reverse-proxy layer (e.g. drop the path component for that prefix).
- Terminate TLS so the token is never on the wire in clear.
- Do not ship access logs to third-party log aggregators without a
  path-scrubbing rule for `/r/`.

### Tamper-detection via log_reason (implemented)

`web.request_view._verify` maps each `InvalidTokenReason` to a distinct
server-side `log_reason` (`_TOKEN_LOG_REASON`):

| reason | log_reason | operational meaning |
|---|---|---|
| EXPIRED | `token_expired` | benign — link past TTL, SB reissues |
| MALFORMED | `token_malformed` | benign — email truncation / copy-paste |
| BAD_SIGNATURE | `token_bad_signature` | **tamper signal** — forged token |

The client always sees the same generic 404 (no disclosure); the split
exists only in the log. A spike of `token_bad_signature` indicates
someone actively forging tokens and is worth alerting on. The token
itself is never in the log line (verified by RT2 + smoke).
